## 1小时1080P视频存储估算

### 采样参数（基于系统默认配置）

| 模态 | 采样率 | 每小时行数估算 | 备注                   |
|------|--------|----------------|----------------------|
| **visual** | 5 fps | 18,000 | 确定值                  |
| **asr** | 语音驱动 | 600-720 | 假设连续语音，每 5 秒 1 chunk |
| **ocr** | 0.5 fps | 1,800 | 取决于内容，假设中密度          |
| **face** | 2 fps 检测 | 300-500 | 取决于内容，假设低密度          |
| **speaker** | ASR 相关 | 600-720 | 与 ASR 块数相当           |

### Milvus 存储计算

#### 估算方法：从向量维度计算（理论值）

**向量字节/行的计算依据**：当前每个嵌入维度存为 float32（4 字节）。
元数据字节/行为各模态 schema 中 scalar 字段（pk、video_id、时间戳等 varchar + int64）的估算合计。

**原始向量 + 元数据大小:**

| 模态 | 行数/小时 | 向量字节/行 (dim × 4B) | 元数据字节/行 | 单行总计 | 小时总计 |
|------|-----------|------------------------|---------------|----------|----------|
| visual | 18,000 | 1152 × 4 = **4,608** | ~300 | 4,908 | 88.3 MiB |
| asr | 720 | 384 × 4 = **1,536** | ~500 | 2,036 | 1.47 MiB |
| ocr | 1,800 | 384 × 4 = **1,536** | ~500 | 2,036 | 3.67 MiB |
| face | 400 | 512 × 4 = **2,048** | ~200 | 2,248 | 0.90 MiB |
| speaker | 720 | 192 × 4 = **768** | ~200 | 968 | 0.70 MiB |
| **总计** | **21,640** | - | - | - | **95.0 MiB** |

**Milvus 生产环境开销分析（HNSW 索引）：**

Milvus 对每个 segment 有两层存储：
1. **binlog**（原始数据）：向量 + scalar 字段，1× 原始大小
2. **HNSW 索引文件**：向量副本（供 ANN 查询）+ 图结构（每节点约 M₀=32 个 int32 邻居指针 = 128 字节）

对 visual（1152-dim）为例：
- binlog 向量：4,608 B/row
- HNSW 向量副本：4,608 B/row
- HNSW 图结构：~128 B/row
- scalar 字段（binlog）：~300 B/row
- **合计约 9,644 B/row ≈ 2.1× 原始向量大小**

加上 RocksDB 元数据、WAL、bloom filter 等，综合系数取 **2.3x**（实测生产环境通常在 1.8–2.3x 之间，2.3x 为保守上限）。

**生产规模 Milvus 存储 = 95.0 MiB × 2.3 ≈ 218 MiB**

**取整保守值: ~230 MiB/hour**

#### 结论

**生产环境 Milvus 存储（1小时1080P视频）: ~230 MiB**

## 30TiB 容量规划

**存储单位说明**：
- 本报告使用 **1024 进制（二进制）** 单位：1 TiB = 1024 GiB = 1,048,576 MiB
- 这是操作系统显示的实际可用容量（Linux `df -h`、Windows 资源管理器）
- 磁盘厂商标注的 "30TB" (SI) 在 OS 中显示为约 27.3 TiB

**30 TiB = 30,720 GiB = 31,457,280 MiB**

---

### 场景 A：仅存储索引数据（无视频文件）

**适用场景:**
- 视频文件存储在外部对象存储（如 OSS/S3）
- 仅本地存储 Milvus 索引 + 元数据
- 通过 URL 引用外部视频

**容量计算:**
```
每小时持久化数据: 230 MiB
30 TiB = 31,457,280 MiB
可存储视频小时数 = 31,457,280 ÷ 230 ≈ 136,771 小时
```

**结果**: **~136,000 小时视频** (约 5,667 天 ≈ 15.5 年连续播放时长)

**额外说明:**
- etcd + minio 固定开销（~167 MiB）在此规模下可忽略
- catalog.sqlite3 增长缓慢，136k 小时 → ~340 GiB，已计入每小时 2.5 MiB

---

### 场景 B：存储索引 + 视频文件（1080P 8Mbps）

**适用场景:**
- 本地一体化存储
- 私有化部署，无外部依赖

**容量计算:**
```
每小时持久化数据: 230 MiB (索引) + 3,435 MiB (视频) = 3,665 MiB
30 TiB = 31,457,280 MiB
可存储视频小时数 = 31,457,280 ÷ 3,665 ≈ 8,583 小时
```

**结果**: **~8,580 小时视频** (约 358 天连续播放时长)

---

### 容量对比表

| 场景 | 每小时存储 | 30TiB 可存储小时数 | 连续播放天数 | 占比分配 |
|------|------------|-------------------|--------------|----------|
| **仅索引** | 230 MiB | 136,771 小时 | 5,667 天 (15.5 年) | 索引 100% |
| **索引+视频** | 3.58 GiB | 8,583 小时 | 358 天 | 索引 6.3% / 视频 93.7% |

## 索引时间计算

**已知条件:**
- 索引速度: **8 分钟 / 1小时视频**
- 即: **0.133 小时索引时间 / 1小时视频内容**

---

### 场景 A：填满 30TiB（仅索引数据）

```
视频小时数: 136,771 小时
索引时间: 136,771 × 8 分钟 = 1,094,168 分钟
        = 18,236 小时
        = 760 天
        ≈ 2.1 年
```

**假设:**
- 单机连续索引，无故障
- GPU/NPU 满载运行
- 实际考虑维护窗口、重试等，约 **2.5-3 年**

---

### 场景 B：填满 30TiB（索引 + 视频）

```
视频小时数: 8,583 小时
索引时间: 8,583 × 8 分钟 = 68,664 分钟
        = 1,144 小时
        = 47.7 天
        ≈ 1.5 个月
```

**假设:**
- 单机连续索引
- 实际考虑维护，约 **1.5-2 个月**

---

### 索引速度优化建议

**当前瓶颈分析（基于 8 分钟/小时）:**
1. **Visual 模态**: 最耗时，5 fps × 3600 = 18k 帧推理
2. **ASR 模态**: Whisper/SenseVoice 推理
3. **Face 模态**: InsightFace 检测 + 特征提取
4. **OCR 模态**: RapidOCR 检测 + 识别
5. **Speaker 模态**: 3D-Speaker 说话人日记化

**加速方案:**
- 多 GPU 并行索引：线性加速比（2 GPU → 4 分钟/小时）
- 降低采样率：visual 5fps → 2fps，减少 60% 计算量
- 模型量化：INT8 推理，加速 2-3x
- 批处理优化：提高 GPU 利用率

**示例**: 4 GPU + 优化后 → **2 分钟/小时视频** → 场景 B 填满 30TB 仅需 **11 天**

## 4K 视频与空间滑窗

### 当前1080P仅有横向（时间轴）分段

`visual.py` 中帧处理的路径是：
1. `read_frames()` 按固定 fps 抽帧
2. `processor / _preprocess_cv2` 将每帧**短边对齐缩放，然后正中心裁剪**至 384×384（SigLIP2 输入尺寸）
3. 每帧出一条 Milvus 行，只有时间维度上的分段（固定窗口或镜头分段）

对于 1080P（1920×1080），中心裁剪保留宽度的 384/1080 ≈ 35%，信息损失尚可接受。

### 4K 帧的信息丢失

4K（3840×2160）→ 短边 2160 对齐后裁中心 384×384，**仅覆盖画面中心约 3% 的像素面积**（384²/（3840×2160）= 1.7%）。边缘字幕、多人画面、分屏内容全部丢弃，无法被检索。

### 纵向（空间）滑窗方案

对每帧按网格划分若干 tile，每个 tile 单独出一条 Milvus 行（携带额外字段 `tile_row`、`tile_col`），并保留一条全帧缩略行（整体语义）：

```
4K 帧 (3840 × 2160)
  ↓ 保守空间分割（3×2 grid，带 20% overlap）
  → tile(0,0)  tile(0,1)  tile(0,2)
    tile(1,0)  tile(1,1)  tile(1,2)
  + 1 条全帧行（down-scale 整帧）
  = 7 条行/帧
```

### 存储规模对比

**视频 1080P（当前，无空间滑窗）:**

| 模态 | 行数/小时 | Milvus 存储/小时 |
|------|-----------|-----------------|
| visual | 18,000 (1帧1行) | ~216 MiB |
| 其他4模态 | ~3,640 | ~14 MiB |
| **合计** | **21,640** | **~230 MiB** |

**视频 4K（加 3×2 空间滑窗）:**

| 模态 | 行数/小时 | Milvus 存储/小时 |
|------|-----------|-----------------|
| visual | 18,000 × 7 = 126,000 (6 tile + 1 全帧) | ~1,512 MiB |
| 其他4模态（不受影响） | ~3,640 | ~14 MiB |
| **合计** | **129,640** | **~1,526 MiB ≈ 1.49 GiB** |

**视频 4K（保守方案，2×2 空间滑窗）:**

| visual 行数/小时 | 18,000 × 5 = 90,000 | Milvus ~1,080 MiB ≈ 1.05 GiB |

### 30TiB 容量（4K 场景）

4K 视频文件大小（H.265/HEVC @ 25 Mbps）：  
25 Mbit/s × 3600 s ÷ 8 bit/byte = 11,250,000,000 bytes ≈ **10,490 MiB ≈ 10.24 GiB/hour**

| 场景 | 每小时存储 | 30TiB 可存小时数 | 连续播放 |
|------|-----------|-----------------|---------|
| 4K 仅索引（3×2 tiling） | 1,526 MiB | ~20,611 小时 | ~859 天 |
| 4K 索引 + 视频 | 1,526 + 10,490 = 12,016 MiB | ~2,618 小时 | ~109 天 |

**对比 1080P（30TiB）:**

| 配置 | 仅索引可存 | 含视频可存 |
|------|-----------|-----------|
| 1080P（无空间滑窗） | 136,771 小时 | 8,583 小时 |
| 4K（3×2 tiling） | 20,611 小时 | 2,618 小时 |
| **倍率** | **1/6.6x** | **1/3.3x** |

> 空间滑窗使 visual Milvus 索引增大 ~6.6x；视频文件本身4K vs 1080P 约 3x（H.265），两者叠加后含视频方案的容量约降至 1080P 方案的 **1/3**。

### 核心发现

1. **索引数据极度高效**: 仅占总存储的 **6%**（含视频场景）
2. **Milvus 向量存储**: 生产规模下约 **230 MB/hour**，小数据集有 27x 开销
3. **视频文件主导**: 1080P @ 8Mbps 占 **94%** 存储空间
4. **30TB 容量**:
   - 仅索引: **13 万小时**（15 年内容）
   - 含视频: **7,800 小时**（326 天内容）
5. **索引时间**:
   - 单机 8 分钟/小时 → 填满 30TB (含视频) 需 **1.5 个月**

### 存储架构建议

#### 方案 1：混合存储（推荐）

```
本地 NVMe SSD (2-4 TB):
  ├── Milvus 向量索引 (热数据，高 IOPS)
  ├── catalog.sqlite3
  └── 最近 7 天视频（快速访问）

对象存储 (S3/OSS):
  └── 全量视频文件 (3.6 GB/hour)
```

**优势**:
- 本地 SSD 存储高价值索引数据
- 对象存储低成本归档视频
- 30TB 对象存储可存 7,800 小时视频

#### 方案 2：本地一体化存储

```
本地 HDD/SSD (30 TB):
  ├── Milvus 索引 (6%)
  └── 视频文件 (94%)
```

**适用**: 私有化部署，无外网访问

---

### Milvus 优化建议

1. **小数据集问题**: 当前 27x 开销不可持续
   - **建议**: 调整 Milvus segment.size 参数（默认 512MB → 64MB）
   - **预期**: 开销降至 5-8x

2. **索引策略**:
   - Visual (18k 行/小时): HNSW, M=16, efConstruction=200
   - 其他模态 (<2k 行/小时): IVF_FLAT, nlist=128（更轻量）

3. **压缩 (Compaction)**:
   - 启用自动 compaction，清理已删除记录
   - 定期 major compaction（每月）

4. **监控指标**:
   - 每个 collection 的 segment 数量
   - rdb_data 实际占用 vs 理论值
   - 查询 P99 延迟

### 数据生命周期管理

```
视频上传 (day 0)
  ↓
索引构建 (8 分钟, 生成 230 MB Milvus 数据)
  ↓
活跃期 (day 0-30): 本地 SSD 缓存视频
  ↓
温数据 (day 30-180): 迁移至对象存储, Milvus 索引保留
  ↓
冷数据 (day 180+): 对象存储归档, Milvus 索引可选保留
  ↓
删除 (可选): 删除 Milvus 向量 + bump asset_version
```

### 扩展性展望

**当索引数据增长到 1PB 时（理论）:**
```
Milvus 数据: 1 PB
对应视频时长: 1 PB / 230 MB = 4,347,826 小时
            = 496 年连续播放
视频文件: 4,347,826 × 3.6 GB = 15.6 PB
```

**结论**: 
- Milvus 索引规模随视频时长线性增长
- 1 PB 索引已可支撑 **半个世纪** 的视频内容检索
- 存储瓶颈永远在视频文件本身，而非索引
